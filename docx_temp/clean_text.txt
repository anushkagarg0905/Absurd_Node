1. Executive Summary
This document defines the complete technical architecture for an AI-Powered OSINT (Open Source Intelligence) Knowledge Graph platform. The system ingests public information from news feeds, RSS sources, Wikipedia, and company websites, then applies a multi-stage NLP and AI pipeline to extract named entities, resolve duplicate records, infer semantic relationships, and persist the results as a traversable Knowledge Graph in Neo4j AuraDB.

The platform exposes intelligence to end users through three primary surfaces: (1) an interactive graph visualization built with React Flow, (2) an intelligence dashboard for aggregate analytics, and (3) AI-generated narrative reports produced by the Gemini API. Metadata, provenance, and document-level indexing are stored in PostgreSQL (Neon).

The definitive technology choices ├втВмтАЭ Next.js / TypeScript / Tailwind CSS / shadcn/ui on the frontend; FastAPI / Python on the backend; spaCy + Hugging Face Transformers + Gemini API for AI/NLP; Neo4j AuraDB + Neon PostgreSQL for persistence; Docker + Render + Vercel for infrastructure ├втВмтАЭ are non-negotiable and are reflected consistently throughout all architectural diagrams and implementation guidance in this document.

This document is organized to allow a senior engineering team to begin implementation immediately. Every major design decision is justified, all database schemas are fully specified, API contracts are defined, and Mermaid diagrams are provided to guide implementation. Alternative approaches are not discussed; the team should treat this document as the single source of engineering truth.

2. Problem Statement
2.1 Limitations of Traditional OSINT Workflows
Traditional OSINT analysis is a manual, analyst-driven process. An analyst searches across dozens of unstructured web sources, reads documents, mentally notes relationships between entities, and assembles findings into flat spreadsheets or word processors. This workflow suffers from four compounding failure modes:
Volume Bottleneck: Human analysts cannot process hundreds of news articles per hour. Critical signal is missed in noise.
Consistency Gaps: Two analysts reading the same corpus will extract different entities, assign different relationship labels, and resolve aliases differently. Results are not reproducible.
Flat Data Model: Spreadsheets cannot natively represent many-to-many relationships. Connections between entities three or four hops away are invisible without manual traversal.
Stale Intelligence: Manual processes are run ad-hoc. There is no mechanism for continuous, near-real-time ingestion and graph enrichment.

2.2 Why Knowledge Graphs Improve Intelligence Analysis
A Knowledge Graph addresses all four failure modes. Entities become nodes with typed relationships as edges. Graph traversal reveals indirect connections automatically. The model is continuously enriched as new data arrives. Deduplication through entity resolution ensures that 'Apple Inc.', 'Apple Computer', and 'AAPL' collapse into a single canonical node.

Beyond data modeling, a graph enables query patterns that are impossible in flat data: shortest-path analysis between two entities, community detection to surface hidden clusters, degree-centrality ranking to identify the most-connected actors. These are first-class features in Neo4j's Cypher query language.

2.3 Project Objectives
Build a fully automated ingestion pipeline that collects public OSINT data continuously from RSS feeds, news APIs, Wikipedia, and company websites.
Apply a production-grade NLP pipeline using spaCy and Hugging Face Transformers for entity extraction and classification.
Use the Gemini API to perform zero-shot relation extraction between entity pairs, assigning typed relationships with confidence scores.
Implement a deterministic entity resolution system that merges duplicate entity records into canonical nodes before writing to the graph.
Maintain a live, queryable Knowledge Graph in Neo4j AuraDB with a well-defined schema of node types and relationship types.
Generate structured, analyst-ready intelligence reports via Gemini API using prompt-engineered templates.
Expose the graph through an interactive React Flow visualization and an intelligence dashboard in a Next.js application.
3. Functional Requirements
3.1 Search Functionality
FR-S1: Users can search for any entity (organization, person, product, location) by name and receive a ranked list of matching canonical nodes from the graph.
FR-S2: Search supports fuzzy matching and alias resolution so that 'Meta', 'Facebook Inc', and 'FB' return the same canonical node.
FR-S3: Search results display node type, confidence score, data sources, and a link to the entity detail view.
FR-S4: Advanced search filters allow scoping by entity type, date range of first/last seen, and minimum relationship count.

3.2 Entity Extraction
FR-E1: The system extracts named entities of types: ORGANIZATION, PERSON, PRODUCT, LOCATION, EVENT, and TECHNOLOGY from ingested text.
FR-E2: Each extracted entity carries a confidence score (0.0├втВмтАЬ1.0) from the NER model and a source document reference.
FR-E3: Entities are classified into canonical types before graph insertion.

3.3 Relationship Extraction
FR-R1: The system identifies typed semantic relationships between entity pairs using Gemini API zero-shot inference.
FR-R2: Supported relationship types: EMPLOYS, ACQUIRED, PARTNERS_WITH, COMPETES_WITH, FOUNDED_BY, LOCATED_IN, INVESTED_IN, SUPPLIES_TO, REGULATES, SUBSIDIARY_OF.
FR-R3: Each relationship carries: source document ID, extraction timestamp, confidence score, and a textual evidence snippet.

3.4 Knowledge Graph Construction
FR-G1: Canonical entities are persisted as nodes in Neo4j with typed labels.
FR-G2: Relationships are persisted as directed edges with typed labels and property maps.
FR-G3: The graph is incrementally updated on each ingestion run without full recomputation.
FR-G4: Duplicate nodes detected post-insertion are merged via MERGE Cypher operations.

3.5 Report Generation
FR-P1: Users can request an AI-generated intelligence report for any entity or pair of entities.
FR-P2: Reports are structured with sections: Executive Summary, Key Relationships, Risk Assessment, Timeline of Events, and Source Index.
FR-P3: Reports are persisted to PostgreSQL and retrievable by report ID.
FR-P4: Report generation uses a Gemini API call with a structured prompt that injects graph context.

3.6 Dashboard Analytics
FR-D1: The dashboard displays aggregate statistics: total entity count by type, relationship count, ingestion rate over time.
FR-D2: A 'Top Entities' widget ranks nodes by degree centrality (number of direct relationships).
FR-D3: A recent-ingestion feed shows the last 20 documents processed and their entity yield.

3.7 Graph Exploration
FR-V1: The graph view renders entities as draggable, interactive nodes with React Flow.
FR-V2: Clicking a node expands its first-degree neighbors up to a configurable depth (default: 2).
FR-V3: Edges display relationship type labels and are color-coded by relationship category.
FR-V4: Users can filter displayed relationships by type or confidence threshold.

4.1 Security Architecture Principles
All inter-service communication uses HTTPS/TLS. Database credentials are injected via environment variables at container startup and never committed to version control. The Gemini API key is stored in Render's secret environment store. Neo4j AuraDB and Neon PostgreSQL both enforce connection-level TLS and IP allowlisting. JWT tokens issued by the backend have a 24-hour expiry and are validated on every request via FastAPI middleware.

PII handling: the NER pipeline flags entities of type PERSON with a pii=true attribute. API endpoints that return PERSON nodes require the requesting JWT to carry a role='analyst' claim. This is enforced in a dedicated FastAPI dependency function, not at the route level, to prevent accidental bypass.

Key Frontend Design Decisions
Server Components are used for all data-fetching pages. Client Components are scoped only to interactive elements (graph canvas, search input). This minimizes client-side JavaScript bundle size.
React Flow is rendered entirely client-side inside a 'use client' boundary wrapper. Graph data is fetched server-side and passed as props to the client component.
shadcn/ui components are copied into the project (not installed as a package) per the shadcn/ui design contract, enabling full customization.
All API calls are made through a typed api-client.ts module that injects the Bearer token from session storage and handles 401 responses by redirecting to login.
The UI design is intentionally decoupled from backend contracts. All backend API shapes are defined by Pydantic response models; the frontend TypeScript types are auto-generated from the OpenAPI schema using openapi-typescript.

6.2 Backend Architecture
The backend is a FastAPI application structured as a modular monolith. All modules share the same Python process and communicate in-process (no internal microservices at MVP). Asynchronous I/O is used throughout using Python's asyncio and the httpx library for outbound HTTP.

7. Data Ingestion Layer
7.1 Collector Architecture
Each collector is an independent async Python class implementing a common CollectorBase abstract interface. The base class enforces the contract: a collect() coroutine that returns List[RawDocument]. This design allows new collectors to be added without modifying the pipeline orchestrator.

# core/collector_base.py

    checksum: str  # SHA-256 of raw_text for dedup

    @abstractmethod

7.2 RSS Collector
The RSS collector uses the feedparser library to parse RSS 2.0 and Atom feeds. Configured feed URLs are stored in the PostgreSQL sources table. The collector fetches all configured feeds concurrently using asyncio.gather, extracts the title, description, link, and pubDate fields, and converts each entry to a RawDocument. HTML tags are stripped from description fields using BeautifulSoup.

7.3 News API Collector
The system integrates with NewsAPI.org (free tier: 100 requests/day) using targeted query terms configured per-deployment. Each API response page is deserialized into RawDocument objects. The collector checkpoints the last-seen article ID in PostgreSQL to avoid reprocessing on subsequent runs.

7.4 Wikipedia Collector
The Wikipedia collector uses the wikipedia-api Python library. It accepts a list of seed entity names (e.g., company names) and fetches their summary sections and the linked entities in the opening two paragraphs. This 'shallow crawl' strategy keeps the document volume manageable while capturing the richest relational signal.

7.5 Website Crawler
The website crawler uses httpx + BeautifulSoup. It is constrained to whitelisted domains only. A depth limit of 2 (parent + one child level) is enforced. The crawler respects robots.txt via the robots_txt_parser library. Rendered JavaScript-heavy pages are not crawled at MVP (Playwright integration is deferred to v1.1).

7.6 Deduplication Strategy
Every RawDocument carries a SHA-256 checksum of its raw_text field. Before a document enters the NLP pipeline, the ingestion service queries PostgreSQL for an existing checksum match. If found, the document is skipped. This is a content-level dedup that correctly handles the same article appearing on multiple sources.

8. NLP Pipeline Design
8.1 Text Preprocessing
Raw text from collectors is preprocessed before NER to improve model accuracy and reduce token cost for Gemini calls. The preprocessing sequence is:
Strip HTML/XML tags using BeautifulSoup(text, 'html.parser').get_text().
Normalize Unicode to NFKC form using unicodedata.normalize('NFKC', text).
Collapse multiple whitespace characters to single space.
Remove boilerplate patterns (cookie notices, navigation menus) via regex blocklist.
Truncate document to 8,000 tokens (├втА░╦Ж 32,000 characters) to fit within spaCy and Gemini context windows.

8.2 Named Entity Recognition
The NER engine uses spaCy's en_core_web_trf model (transformer-backed, based on RoBERTa). This model is chosen over the smaller en_core_web_sm/md because it achieves F1 scores of 89├втВмтАЬ91 on OntoNotes 5.0, which contains all the entity types this system requires.

# services/nlp_pipeline.py (NER section)
import spacy
from spacy.tokens import Doc

nlp = spacy.load('en_core_web_trf')
ACCEPTED_LABELS = {'ORG', 'PERSON', 'PRODUCT', 'GPE', 'LOC', 'EVENT', 'LAW'}
LABEL_MAP = {
    'ORG': 'ORGANIZATION', 'GPE': 'LOCATION', 'LOC': 'LOCATION',
    'PERSON': 'PERSON', 'PRODUCT': 'PRODUCT', 'EVENT': 'EVENT', 'LAW': 'REGULATION'
}

def extract_entities(text: str) -> list[dict]:
    doc: Doc = nlp(text)
    entities = []
    for ent in doc.ents:
        if ent.label_ not in ACCEPTED_LABELS:
            continue
        confidence = ent._.trf_data.tensors[0].max().item()  # approx confidence
        if confidence < 0.65:
            continue
        entities.append({
            'text': ent.text, 'type': LABEL_MAP[ent.label_],
            'start_char': ent.start_char, 'end_char': ent.end_char,
            'confidence': round(confidence, 4),
            'sentence': ent.sent.text.strip()
        })
    return entities

8.3 Relation Extraction via Gemini API
After entity extraction, the pipeline generates entity pairs within each sentence (co-occurring entities in the same sentence). For each sentence containing two or more entities, a Gemini prompt is constructed and submitted. The prompt uses a fixed structured output schema to ensure parseable JSON responses.

The prompt template is:

RELATION_EXTRACTION_PROMPT = '''
You are an expert information extraction system. Given the sentence below and
the identified named entities, extract semantic relationships between entity pairs.

SENTENCE: {sentence}
ENTITIES: {entities_json}

Return ONLY a JSON array. Each element must have:
  - subject: string (entity text)
  - predicate: one of [EMPLOYS, ACQUIRED, PARTNERS_WITH, COMPETES_WITH,
      FOUNDED_BY, LOCATED_IN, INVESTED_IN, SUPPLIES_TO, REGULATES,
      SUBSIDIARY_OF, MENTIONS]
  - object: string (entity text)
  - confidence: float 0.0-1.0
  - evidence: the exact substring that supports this relationship

If no clear relationship exists, return an empty array [].
Return ONLY the JSON array, no explanation, no markdown.
'''

8.4 Confidence Scoring
Two independent confidence scores flow through the system. NER confidence is the softmax probability from the spaCy transformer's final CLS token projection. Relation confidence is the value returned by Gemini in the structured output. The final entity record persisted to Neo4j carries both scores independently. Relationships with confidence below 0.55 are written to Neo4j but flagged with a low_confidence=true property and excluded from the default graph visualization.
9. Entity Resolution System
9.1 Why Custom Entity Resolution
Entity resolution (ER) is the process of determining that two extracted entity mentions refer to the same real-world entity. Off-the-shelf ER services (Diffbot, entity-fishing) introduce external dependencies and cost. A deterministic, rule-plus-embedding approach is more auditable, less expensive, and sufficient for the entity volume at MVP.

9.3 Text Normalization Rules
Lowercase the entity text.
Remove legal suffixes: 'Inc', 'LLC', 'Ltd', 'Corp', 'Co.', 'plc', 'GmbH' (suffix list in config).
Remove possessive apostrophes ('Apple's' ├втАатАЩ 'Apple').
Strip punctuation except internal hyphens.
Normalize whitespace.

9.4 Embedding-Based Similarity
When phonetic matching yields a candidate but is not definitive, the system computes cosine similarity between sentence-transformers embeddings of the normalized entity names. The model used is sentence-transformers/all-MiniLM-L6-v2, which produces 384-dimensional embeddings at low computational cost. The similarity threshold of 0.88 was chosen empirically to minimize false positives (merging distinct entities) while catching genuine aliases.

# services/entity_resolution.py
from sentence_transformers import SentenceTransformer
from sklearn.metrics.pairwise import cosine_similarity
import numpy as np

model = SentenceTransformer('sentence-transformers/all-MiniLM-L6-v2')
SIMILARITY_THRESHOLD = 0.88

def are_same_entity(name_a: str, name_b: str) -> bool:
    embeddings = model.encode([name_a, name_b])
    sim = cosine_similarity([embeddings[0]], [embeddings[1]])[0][0]
    return float(sim) >= SIMILARITY_THRESHOLD

10.3 Graph Traversal Patterns
The following Cypher patterns are the primary queries executed by the graph service. Each is parameterized and indexed appropriately.

// Pattern 1: Find all relationships within N hops of an entity
MATCH path = (e:Organization {canonical_id: $id})-[*1..2]-(neighbor)
WHERE NOT neighbor:Source
RETURN path LIMIT 100

// Pattern 2: Shortest path between two entities
MATCH (a {canonical_id: $id_a}), (b {canonical_id: $id_b})
MATCH path = shortestPath((a)-[*..6]-(b))
RETURN path

// Pattern 3: Top entities by degree centrality
MATCH (n)-[r]-()
WHERE NOT n:Source
RETURN n.canonical_id, n.name, labels(n)[0] AS type,
       count(r) AS degree
ORDER BY degree DESC LIMIT 20

// Pattern 4: All organizations in a supply chain
MATCH path = (org:Organization {canonical_id: $id})
             -[:SUPPLIES_TO*1..4]->(customer:Organization)
RETURN path
11. Neo4j AuraDB Design
11.1 Constraints
All constraints are applied at database initialization via a Cypher migration script run on first boot. Constraints enforce uniqueness of canonical_id across all node labels, which is the primary lookup key used by the entity resolution engine.

// constraints.cypher ├втВмтАЭ run once at initialization
CREATE CONSTRAINT org_id_unique IF NOT EXISTS
  FOR (o:Organization) REQUIRE o.canonical_id IS UNIQUE;

CREATE CONSTRAINT person_id_unique IF NOT EXISTS
  FOR (p:Person) REQUIRE p.canonical_id IS UNIQUE;

CREATE CONSTRAINT product_id_unique IF NOT EXISTS
  FOR (pr:Product) REQUIRE pr.canonical_id IS UNIQUE;

CREATE CONSTRAINT location_id_unique IF NOT EXISTS
  FOR (l:Location) REQUIRE l.canonical_id IS UNIQUE;

CREATE CONSTRAINT source_id_unique IF NOT EXISTS
  FOR (s:Source) REQUIRE s.source_id IS UNIQUE;

11.2 Indexes
// indexes.cypher
// Full-text search index across all entity names
CREATE FULLTEXT INDEX entity_name_fulltext IF NOT EXISTS
  FOR (n:Organization|Person|Product|Location|Technology)
  ON EACH [n.name];

// Range index on last_seen for recency queries
CREATE INDEX org_last_seen IF NOT EXISTS
  FOR (o:Organization) ON (o.last_seen);

// Composite index for type + name lookup
CREATE INDEX org_name IF NOT EXISTS
  FOR (o:Organization) ON (o.name);

CREATE INDEX person_name IF NOT EXISTS
  FOR (p:Person) ON (p.name);

11.3 MERGE Write Pattern
All writes to Neo4j use MERGE to ensure idempotency. The entity resolution engine produces canonical_id values before the write; MERGE on canonical_id is effectively always an O(1) lookup due to the UNIQUE constraint index.

// Canonical node upsert (called from graph_service.py)
MERGE (o:Organization {canonical_id: $canonical_id})
ON CREATE SET
  o.name = $name,
  o.aliases = $aliases,
  o.first_seen = datetime($first_seen),
  o.last_seen = datetime($last_seen),
  o.mention_count = 1
ON MATCH SET
  o.last_seen = datetime($last_seen),
  o.mention_count = o.mention_count + 1,
  o.aliases = apoc.coll.union(o.aliases, $aliases)
RETURN o

// Relationship upsert
MATCH (a {canonical_id: $subject_id})
MATCH (b {canonical_id: $object_id})
MERGE (a)-[r:ACQUIRED]->(b)
ON CREATE SET r.confidence = $confidence, r.source_id = $source_id,
  r.first_seen = datetime($ts)
ON MATCH SET r.last_seen = datetime($ts),
  r.confidence = ($confidence + r.confidence) / 2.0

13.3 Prompt Engineering Strategy
The Gemini prompt is assembled by the ReportPromptBuilder class. The builder injects: (1) entity metadata, (2) a serialized adjacency list of the 2-hop subgraph, (3) the 10 most recent source titles and excerpts, and (4) the report type instruction. All sections are clearly delimited with XML-style tags to prevent prompt injection from source document content.

REPORT_SYSTEM_PROMPT = '''
You are an expert intelligence analyst. Generate a structured intelligence report
in JSON format based only on the provided graph context and source excerpts.
Do not hallucinate facts not present in the provided data.
Return ONLY valid JSON matching the specified schema. No markdown, no explanation.
'''

REPORT_USER_TEMPLATE = '''
<report_type>{report_type}</report_type>

<graph_context>{subgraph_json}</graph_context>
<source_excerpts>{excerpts}</source_excerpts>

13.4 Cost Control
Each report generation call is logged with prompt_tokens and completion_tokens to the intelligence_reports table. A rate limiter (using Redis token bucket) caps report generation at 50 reports per hour system-wide. The context passed to Gemini is capped at 12,000 tokens: 8,000 for graph context and 4,000 for source excerpts. The Gemini 1.5 Flash model is used (not Pro) for its 3x lower cost at acceptable quality for OSINT reporting tasks.
14. API Design
14.1 API Architecture Principles
All endpoints return consistent JSON envelopes with data, meta, and error fields.
Pagination uses cursor-based pagination (not offset-based) to support large graph result sets.
All mutation endpoints (POST/PUT/DELETE) require a Bearer JWT with role-appropriate claims.
OpenAPI 3.1 documentation is auto-generated at /docs and /redoc.
Request validation is handled by Pydantic models at the route parameter level.

15.4 CI/CD Pipeline
GitHub Actions is used for CI. On every PR to main: run pytest (backend), run eslint + tsc (frontend), build Docker image.
Render auto-deploys on merge to main via GitHub integration. Zero-downtime deploy using Render's rolling restart.
Vercel auto-deploys on merge to main via GitHub integration. Preview deployments are created for all PRs.
Database migrations (Alembic for PostgreSQL, Cypher scripts for Neo4j) are run as a pre-deploy step in the Render deploy script, not in the application startup code.

16.2 React Flow Graph Component
The graph visualizer is the core interactive component. It receives serialized Neo4j subgraph data from the /entities/{id}/graph endpoint and transforms it into React Flow nodes and edges. Node types are color-coded; edge labels show relationship types.

// components/graph/KnowledgeGraph.tsx
'use client';
import { ReactFlow, Background, Controls, MiniMap,
         useNodesState, useEdgesState } from '@xyflow/react';
import { transformSubgraphToFlow } from '@/lib/graph-utils';
import { EntityNode } from './EntityNode';
import { RelationshipEdge } from './RelationshipEdge';

const nodeTypes = { entity: EntityNode };
const edgeTypes = { relationship: RelationshipEdge };

export function KnowledgeGraph({ subgraph }) {
  const { nodes: initNodes, edges: initEdges } = transformSubgraphToFlow(subgraph);
  const [nodes, , onNodesChange] = useNodesState(initNodes);
  const [edges, , onEdgesChange] = useEdgesState(initEdges);

  return (
    <div style={{ width: '100%', height: '600px' }}>
      <ReactFlow nodes={nodes} edges={edges}
                 nodeTypes={nodeTypes} edgeTypes={edgeTypes}
                 onNodesChange={onNodesChange}
                 onEdgesChange={onEdgesChange} fitView>
        <Background /> <Controls /> <MiniMap />
      </ReactFlow>
    </div>
  );
}

16.3 Data Fetching Strategy
SWR (stale-while-revalidate) is used for all client-side data fetching. This provides automatic revalidation, error handling, and loading states without writing boilerplate. Server Components are used for the initial data load to prevent layout shift.

// hooks/useGraph.ts
import useSWR from 'swr';
import { fetcher } from '@/lib/api-client';

export function useEntityGraph(entityId: string, depth: number = 2) {
  const { data, error, isLoading } = useSWR(
    entityId ? `/entities/${entityId}/graph?depth=${depth}` : null,
    fetcher
  );
  return { subgraph: data?.data, error, isLoading };
}

17. Testing Strategy

19. Design Decision Log
Every major architectural decision is logged here with its rationale. This log is the authoritative reference for 'why did we build it this way' questions during implementation and future code reviews.

Decision ID
Decision
Rationale
Rejected Alternative
DD-01
Use spaCy en_core_web_trf for NER (not fine-tuned custom model)
Fine-tuning requires labeled OSINT dataset (weeks to build). en_core_web_trf achieves >89 F1 on OntoNotes entity types. Suitable for MVP.
Fine-tuning BERT on custom corpus ├втВмтАЭ rejected due to labeling effort
DD-02
Use Gemini 1.5 Flash for relation extraction (not a fine-tuned model)
Zero-shot relation extraction with a well-engineered prompt avoids fine-tuning overhead. Flash model is 3x cheaper than Pro.
Fine-tuning a HF classification model on RE datasets ├втВмтАЭ rejected due to dataset acquisition cost
DD-03
Entity resolution without a dedicated ER service (custom pipeline)
Dedupe/Zingg SaaS adds cost and data egress. The phonetic + embedding approach is deterministic, auditable, and free.
Dedupe.io or Zingg ├втВмтАЭ rejected due to cost and external dependency
DD-04
No microservices at MVP (modular monolith backend)
Microservices add operational overhead without benefit at MVP scale. The modular monolith is split into independent modules by domain and can be extracted to services in v2.
Microservices from day one ├втВмтАЭ rejected due to DevOps complexity overhead
DD-05
Cursor-based pagination over offset-based
Offset pagination breaks when new records are inserted between pages (a common scenario in a live ingestion system). Cursors are stable.
LIMIT/OFFSET pagination ├втВмтАЭ rejected due to instability under concurrent writes
DD-06
React Flow over D3.js or Cytoscape.js for graph rendering
React Flow is React-native with hooks, drag-and-drop, custom node types as JSX. D3 and Cytoscape require imperative DOM manipulation that conflicts with React's rendering model.
D3.js direct integration ├втВмтАЭ rejected due to React reconciliation conflicts
DD-07
Neon PostgreSQL (serverless) over self-hosted PostgreSQL
Neon's serverless tier scales to zero when idle, eliminating cost during development pauses. It is fully compatible with standard PostgreSQL drivers.
Self-hosted PostgreSQL on Render ├втВмтАЭ rejected due to persistence management overhead
DD-08
Use SHA-256 document checksum for dedup (not URL-based)
The same article is published on multiple domains (syndication). URL-based dedup would process the same content N times. Content hash is canonical.
URL-based deduplication ├втВмтАЭ rejected due to syndication problem
20. Glossary
Term
Definition
OSINT
Open Source Intelligence ├втВмтАЭ intelligence derived from publicly available information
NER
Named Entity Recognition ├втВмтАЭ NLP task of identifying named entities in text and classifying them into predefined categories
Relation Extraction
NLP task of identifying semantic relationships between entity pairs in a text passage
Entity Resolution (ER)
Process of determining that two or more entity mentions refer to the same real-world entity; also called record linkage or deduplication
Canonical Entity
The single, authoritative record for a real-world entity after all aliases have been resolved and merged
Knowledge Graph
A graph database that represents real-world entities as nodes and semantic relationships as typed edges
Neo4j AuraDB
Managed cloud-hosted Neo4j graph database service provided by Neo4j Inc.
Neon
Serverless PostgreSQL database service that scales to zero when idle
Cypher
Neo4j's declarative graph query language, analogous to SQL for graph databases
React Flow
Open-source React library for building interactive node-based graph and diagram editors
spaCy
Industrial-strength NLP library for Python. en_core_web_trf is its transformer-based English model
Gemini API
Google's large language model API. Gemini 1.5 Flash is used for relation extraction and report generation
JWT
JSON Web Token ├втВмтАЭ compact, URL-safe means of representing claims between two parties for authentication
MERGE (Cypher)
Cypher operation that either matches an existing node/relationship or creates it; the primary idempotent write pattern for this system
Phonetic Encoding
Algorithm that produces a code representing how a name sounds, used for alias matching (soundex, metaphone)
Sentence Transformers
HuggingFace library for computing dense vector embeddings from text, used in embedding-based entity resolution
